Note on cmd.Dir: always set to cwd (the repo root). The sandflox binary's main.go calls
`os.Getwd()` to resolve projectDir, so the subprocess must run from the repo root to pick up
the repo's policy.toml and .flox/env.json.

Note on separating stdout from stderr: Phase 2 uses `cmd.CombinedOutput()` which mixes streams.
For Phase 3 shell enforcement tests, we often need to assert specific BLOCKED messages appear on
STDERR specifically (per SHELL-08 convention). Use the Phase 2 pattern where separate streams
matter:
```go
var stdoutBuf, stderrBuf bytes.Buffer
cmd.Stdout = &stdoutBuf
cmd.Stderr = &stderrBuf
err := cmd.Run()
stdout := stdoutBuf.String()
stderr := stderrBuf.String()
exitCode := 0
if ee, ok := err.(*exec.ExitError); ok { exitCode = ee.ExitCode() }
```

Note on exit codes through flox activate: a command that runs via `sandflox -- CMD` goes through
sandbox-exec → flox activate → bash -c 'source entrypoint && exec "$@"'. When CMD exits 126
(our BLOCKED exit code from armor functions), bash exits 126, flox activate exits 126, and the
sandflox binary's syscall.Exec-replaced process exits 126. Assertion: `exitCode == 126` for
armor-function hits; Python raises PermissionError which propagates as a Python traceback with
exit 1.

<reference_patterns>
03-RESEARCH.md Pitfall 7: do NOT use syscall.Exec from tests. Always use exec.CommandContext.

The existing exec_integration_test.go's TestBuiltBinaryWrapsCommand is a working proof that the
build-once + invoke + assert pattern works. All Phase 3 tests follow the same skeleton:

```go
func TestShellEnforces_<thing>(t *testing.T) {
    skipIfNoSandboxExec(t)  // Phase 2 helper
    // … flox / .flox/env.json / policy.toml existence checks …

    // Build binary once per test (t.TempDir isolates each).
    cwd, _ := os.Getwd()
    binPath := filepath.Join(t.TempDir(), "sandflox")
    build := exec.Command("go", "build", "-o", binPath, ".")
    build.Dir = cwd
    if out, err := build.CombinedOutput(); err != nil {
        t.Fatalf("build failed: %v\n%s", err, out)
    }

    // Invoke with probe.
    ctx, cancel := context.WithTimeout(context.Background(), 60*time.Second)
    defer cancel()
    cmd := exec.CommandContext(ctx, binPath, "--", "/bin/bash", "-c", probeScript)
    cmd.Dir = cwd
    var so, se bytes.Buffer
    cmd.Stdout = &so
    cmd.Stderr = &se
    err := cmd.Run()
    exitCode := 0
    if ee, ok := err.(*exec.ExitError); ok { exitCode = ee.ExitCode() }

    // Assert per requirement.
}
```

A single `runSandflox` helper declared at the top of the file de-duplicates this skeleton.

Policy.toml note: the test suite runs inside the sandflox repo itself, which has a real
policy.toml. Tests that need a different policy (e.g. net="unrestricted" to prove curl is
PRESENT) must either (a) skip themselves when the repo policy doesn't match, or (b) temporarily
swap the policy via the `--policy <path>` flag. Option (b) is cleaner — sandflox supports
`--policy path/to/other.toml` per Phase 1.

Note on multiple binary builds: building the sandflox binary takes ~3-5s on a warm system.
Rather than building 9 times, use `TestMain` to build once into a package-level var that all
tests reuse. But `TestMain` adds complexity and makes test skip-flow awkward when flox/env.json
is absent. Compromise: build once per test via `sync.OnceFunc` pattern, OR just accept the
~30-45s total build cost across all tests (each test still completes in <10s wall). Pick the
simpler approach: build-once-per-test. If total runtime bothers, TestMain can be added later.

Actually cleaner: use `sync.Once` + a package-level `var sharedBinPath string` so the first test
builds and subsequent tests reuse. Implement this in the runSandflox helper.
