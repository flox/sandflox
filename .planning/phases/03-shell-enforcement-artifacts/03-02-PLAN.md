```go
func buildSandboxExecArgv(sbplPath, projectDir, home, floxCachePath, floxAbsPath, entrypointPath string, userArgs []string) []string

// Interactive (len(userArgs)==0):
// ["sandbox-exec", "-f", sbplPath, "-D", "PROJECT=...", "-D", "HOME=...", "-D", "FLOX_CACHE=...",
//  floxAbsPath, "activate", "--", "bash", "--rcfile", entrypointPath, "-i"]
// Length: 15 elements.

// Non-interactive (len(userArgs)>0), e.g. userArgs=["echo","hello"]:
// ["sandbox-exec", "-f", sbplPath, "-D", "PROJECT=...", "-D", "HOME=...", "-D", "FLOX_CACHE=...",
//  floxAbsPath, "activate", "--", "bash", "-c", "source " + shellquote(entrypointPath) + ` && exec "$@"`, "--",
//  "echo", "hello"]
// Length: 16 + len(userArgs).
// CRITICAL: the inner "source <entrypoint> && exec \"$@\"" string is the bash-script ARGUMENT to -c.
// The "--" AFTER the -c string is for bash's own argv boundary ($0 is skipped by convention — we
// use "--" so that "echo" lands as $1 not $0). This mirrors sandflox.bash:459-463 and D-02.
```

Note on the bash -c payload literal: use Go's backtick raw string or a regular string with escaped
quotes. The shell script itself contains a literal `"$@"` — bash consumes the double quotes at shell
runtime, so Go passes the bytes `source /abs/path/entrypoint.sh && exec "$@"` as one argv element.
Using backticks avoids escape hell: `` `source ` + entrypointPath + ` && exec "$@"` `` — but
entrypointPath must be shell-safe. Since cacheDir is project-relative and projectDir comes from
os.Getwd / --policy resolution, paths containing single quotes are pathological but not impossible.
Use the shellquote helper from shell.go to wrap entrypointPath: `"source " + shellquote(entrypointPath) + ` && exec "$@"``.
This makes the payload robust against any path (Pitfall 1 from 03-RESEARCH.md).

shellquote is already defined (lowercase, unexported) in shell.go — same package, direct call.
