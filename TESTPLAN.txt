sandflox v1.0 — Manual Test Plan

  Prerequisites

  - Fresh terminal, outside the sandflox project
  - flox installed (1.10+)
  - A directory with policy.toml available
  - Flox environment must include supporting packages that sandflox will whitelist.
    The FloxHub package provides only the sandflox binary; tools like bash, coreutils,
    python3, jq, curl, git must be installed separately:
      flox install bash coreutils python3 jq curl git gnugrep gnused gawk findutils diffutils file
  - Copy requisites files (requisites.txt, requisites-minimal.txt, requisites-full.txt)
    from the sandflox source repo into the test project directory alongside policy.toml

  1. Install from FloxHub

  cd /tmp && mkdir sfx-test && cd sfx-test && flox init
  flox install 8BitTacoSupreme/sandflox
  flox activate -- command -v sandflox  # expect: /nix/store/... path

  2. Validate (CMD-01)

  cp ~/sandflox/policy.toml .
  flox activate -- sandflox validate -policy policy.toml
  # expect: profile, fs mode, net mode, denied paths, tool count — no sandbox launched

  3. Validate with debug (CORE-05/06)

  flox activate -- sandflox validate -policy policy.toml -debug
  # expect: [sandflox] prefixed diagnostics, SBPL rule count, cache artifacts written

  4. Default sandbox launch (KERN-01, SHELL-01..03)

  flox activate -- sandflox -policy policy.toml
  # inside sandbox:
  echo $PATH              # only sandflox symlink bin dir
  command -v pip           # not found
  pip install foo          # [sandflox] BLOCKED: pip is not available. Environment is immutable.
  exit

  5. Non-interactive command (KERN-06)

  flox activate -- sandflox -policy policy.toml -- echo "hello from sandbox"
  # expect: "hello from sandbox", clean exit

  6. Network blocking (KERN-07/08)

  # with policy.toml network.mode = "blocked":
  flox activate -- sandflox -policy policy.toml -- curl https://example.com
  # expect: kernel-level failure (Operation not permitted)

  7. Filesystem enforcement (KERN-02/03, SHELL-04)

  flox activate -- sandflox -policy policy.toml
  # inside sandbox:
  tee /tmp/allowed.txt <<< "test"  # should succeed (workspace writable)
  cp /etc/hosts /tmp/stolen        # [sandflox] BLOCKED: write to "/etc/hosts" outside workspace policy
  exit

  8. Env sanitization (SEC-01/02)

  export AWS_SECRET_ACCESS_KEY=leaked
  flox activate -- sandflox -policy policy.toml -- printenv AWS_SECRET_ACCESS_KEY
  # expect: empty (scrubbed)
  flox activate -- sandflox -policy policy.toml -- printenv HOME
  # expect: /Users/jhogan (allowlisted)
  unset AWS_SECRET_ACCESS_KEY

  9. Profile override (CORE-03)

  flox activate -- sandflox -policy policy.toml -profile minimal
  # expect: reduced tool set (~28 tools)

  10. Status from inside sandbox (CMD-02)

  flox activate -- sandflox -policy policy.toml
  # inside sandbox:
  sandflox status
  # expect: Profile: default | Network: blocked | Filesystem: workspace
  # expect: Tools: N | Denied paths: 5
  exit

  11. Elevate (CMD-03)

  flox activate   # plain flox, no sandbox
  sandflox elevate -policy ~/sandflox/policy.toml
  # expect: re-execs under sandbox-exec
  sandflox elevate -policy ~/sandflox/policy.toml
  # expect: "Already sandboxed -- nothing to do."
  exit

  12. Clean up

  # Ensure no sandbox processes are still running (active flox sessions
  # hold locks on .flox/run/ and will prevent deletion)
  ps aux | grep -E 'sandbox-exec|sandflox' | grep -v grep
  # If any processes appear, exit them first, then:
  cd /tmp && rm -rf sfx-test

  Pass Criteria

  All 11 scenarios behave as described. No kernel panics, no orphan processes, no credential leaks.

  flox-sbx Consumer Environment Tests

  These tests validate the two-tier enforcement model:
    Tier 1 — Shell-level: active on `flox activate` (default posture)
    Tier 2 — Kernel-level: added by `sandflox elevate` (shields up)

  Workspace mode (filesystem = "workspace") applies at BOTH tiers:
    Shell: fs-filter.sh wraps write commands -> exit 126 + BLOCKED message
    Kernel: SBPL deny file-write* -> OS EPERM

  Note: Shell redirections (>, >>, |) are handled by bash internally and
  cannot be intercepted by fs-filter. Kernel tier blocks them at the
  syscall level. This is why `sandflox elevate` exists.

  Prerequisites (flox-sbx)

  - Fresh terminal
  - flox 1.10+ installed
  - aarch64-darwin (ARM Mac) — sandflox binary is currently published for
    ARM Macs only. x86_64-darwin gets shell enforcement without the binary.
  - No policy.toml needed — sandflox uses an embedded default policy when
    no policy.toml is found on disk.

  Setup (flox-sbx)

  mkdir /tmp/sfx-consumer-test && cd /tmp/sfx-consumer-test
  flox pull 8BitTacoSupreme/flox-sbx
  # expect: "Pulled 8BitTacoSupreme/flox-sbx from https://hub.flox.dev/"

  ── Tier 1: Shell-Level (Default Posture) ─────────────────

  13. Activate and verify PATH wipe

  flox activate
  echo $PATH                 # expect: .../sandflox/bin (symlink dir only)
  command -v ls              # expect: path inside sandflox/bin, NOT /usr/bin/ls
  command -v sudo            # expect: empty / exit 1 (not in requisites)
  command -v bash            # expect: found (in requisites)
  env | grep SANDFLOX        # expect: SANDFLOX_ENABLED=1, SANDFLOX_MODE=enforced

  14. Function armor blocks package managers

  pip install requests       # expect: [sandflox] BLOCKED: pip is not available. Environment is immutable.
                             # exit code: 126
  npm install express        # expect: [sandflox] BLOCKED: npm is not available. Environment is immutable.
  brew install wget          # expect: [sandflox] BLOCKED: brew is not available. Environment is immutable.
  docker run hello-world     # expect: [sandflox] BLOCKED: docker is not available. Environment is immutable.
  flox install curl          # expect: [sandflox] BLOCKED: flox is not available. Environment is immutable.
  nix-shell -p wget          # expect: [sandflox] BLOCKED: nix-shell is not available. Environment is immutable.
  go get github.com/foo/bar  # expect: [sandflox] BLOCKED: go is not available. Environment is immutable.

  15. Filesystem write enforcement (workspace mode — shell tier)

  # Note: fs-filter wraps 8 commands: cp, mv, mkdir, rm, rmdir, ln, chmod, tee.
  # `touch` and shell redirections (>) bypass fs-filter entirely.
  # Use wrapped commands for reliable enforcement tests.
  tee /tmp/test.txt <<< "test"          # expect: succeeds (writable path)
  mkdir /tmp/sfx-test-dir               # expect: succeeds (writable path)
  tee ./testfile <<< "test"             # expect: succeeds (project dir is writable)
  cp ./testfile .flox/env/x             # expect: [sandflox] BLOCKED: ".flox/env/x" is read-only by policy
  tee policy.toml <<< "bad"             # expect: [sandflox] BLOCKED: "policy.toml" is read-only by policy
                                        #   (only if policy.toml exists; otherwise blocked as outside workspace)
  mkdir ~/.ssh/pwned                    # expect: [sandflox] BLOCKED: write to "~/.ssh/pwned" is denied by policy
  cp /etc/hosts /usr/local/stolen       # expect: [sandflox] BLOCKED: write to "/usr/local/stolen" outside workspace policy
  # clean up:
  rm -f ./testfile /tmp/test.txt && rmdir /tmp/sfx-test-dir

  16. Python enforcement

  python3 -c "import ensurepip"
  # expect: SystemExit("[sandflox] BLOCKED: ensurepip is disabled in this sandbox")

  python3 -c "open('/etc/passwd', 'w')"
  # expect: PermissionError("[sandflox] BLOCKED: write to '/etc/passwd' outside workspace policy")

  python3 -c "open('/tmp/ok.txt', 'w').write('hello')"
  # expect: succeeds (/tmp is writable)
  rm -f /tmp/ok.txt

  17. curl removed when network=blocked

  command -v curl            # expect: empty / exit 1 (removed from symlink bin)

  18. Breadcrumb cleanup

  echo $FLOX_ENV_PROJECT     # expect: empty (unset)
  echo $FLOX_ENV_DIRS        # expect: empty (unset)
  echo $FLOX_PATH_PATCHED    # expect: empty (unset)

  19. sandflox status from inside shell enforcement

  sandflox status
  # expect: [sandflox] Profile: default | Network: blocked | Filesystem: workspace
  # expect: [sandflox] Tools: 57 | Denied paths: 5

  exit  # leave flox-sbx shell

  ── Tier 2: Kernel-Level (Shields Up) ─────────────────────

  20. Elevate to kernel enforcement

  flox activate
  sandflox elevate
  # expect: "[sandflox] WARNING: no policy.toml -- using embedded default"
  # expect: "[sandflox] Profile: default | Network: blocked | Filesystem: workspace"
  # expect: "[sandflox] Elevating to sandboxed shell (sandbox-exec)"
  # expect: "[sandflox] Sandbox active: N tools (profile: default)"

  21. Kernel network blocking (not just PATH removal)

  # Inside elevated shell:
  # Even if an agent compiles a binary that opens a socket, it's blocked:
  python3 -c "import urllib.request; urllib.request.urlopen('http://example.com')"
  # expect: kernel EPERM / socket error (not just "curl not found")
  # The key difference: shell-only removes curl from PATH but can't block
  # raw socket() syscalls. Kernel tier denies the syscall itself.

  22. Kernel denied-path read blocking

  cat ~/.ssh/id_rsa          # expect: EPERM (kernel blocks even reads)
  cat ~/.aws/credentials     # expect: EPERM
  # Shell-only CANNOT block reads — only kernel tier denies file-read*.

  23. Kernel workspace write enforcement (EPERM)

  tee /usr/local/pwned <<< x  # expect: EPERM (not [sandflox] BLOCKED:)
  tee ~/.bashrc <<< bad        # expect: EPERM (outside workspace)
  tee ./kernel-ok.txt <<< ok   # expect: succeeds (project dir writable)
  rm -f ./kernel-ok.txt
  # Key difference from shell: kernel returns OS-level EPERM with no
  # custom message, while shell returns exit 126 with [sandflox] BLOCKED:

  24. Env sanitization at kernel exec

  # From OUTSIDE flox-sbx, set a credential:
  exit  # leave elevated shell
  exit  # leave flox shell
  export AWS_SECRET_ACCESS_KEY=leaked
  flox activate
  sandflox elevate
  printenv AWS_SECRET_ACCESS_KEY  # expect: empty (scrubbed at exec)
  printenv HOME                   # expect: present (allowlisted)
  printenv SANDFLOX_ENABLED       # expect: 1 (allowlisted)
  unset AWS_SECRET_ACCESS_KEY
  exit  # leave elevated shell
  exit  # leave flox shell

  ── Cross-Tier Tests ──────────────────────────────────────

  25. Workspace enforcement at BOTH tiers

  flox activate
  # Shell tier:
  cp /etc/hosts ~/.ssh/stolen     # expect: [sandflox] BLOCKED (denied path)
  # Elevate:
  sandflox elevate
  # Kernel tier (same policy, different enforcement mechanism):
  cp /etc/hosts ~/.ssh/stolen     # expect: BLOCKED (kernel EPERM on denied path)
  tee .flox/env/pwned <<< x       # expect: BLOCKED (read-only at both tiers)
  exit
  exit

  26. Elevate re-entry detection

  flox activate
  sandflox elevate
  # Inside elevated shell, try to elevate again:
  sandflox elevate
  # expect: "[sandflox] Already sandboxed -- nothing to do." exit 0
  exit
  exit

  27. Graceful degradation (no policy.toml)

  # This test is now the DEFAULT consumer flow. flox-sbx uses the embedded
  # default policy when no policy.toml exists on disk.
  mkdir /tmp/sfx-degrade-test && cd /tmp/sfx-degrade-test
  flox pull 8BitTacoSupreme/flox-sbx
  flox activate
  # expect: "[sandflox] WARNING: no policy.toml -- using embedded default"
  # expect: "[sandflox] Prepared: 57 tools | default | net:blocked fs:workspace"
  # expect: "[sandflox] Sandbox active: 57 tools (profile: default)"
  pip install foo       # expect: [sandflox] BLOCKED (function armor)
  echo $PATH            # expect: .../sandflox/bin (symlink dir)
  command -v curl       # expect: empty (network blocked, curl removed)
  sandflox status       # expect: Profile: default | Network: blocked | Filesystem: workspace
  exit
  cd /tmp && rm -rf sfx-degrade-test

  28. Clean up (flox-sbx)

  # Ensure no sandbox processes remain:
  ps aux | grep -E 'sandbox-exec|sandflox' | grep -v grep
  # If any appear, exit them first.
  cd /tmp && rm -rf sfx-consumer-test

  Pass Criteria (flox-sbx)

  All tests 13-28 behave as described.
  - Shell-only (13-19): agents blocked from reaching package managers,
    writing outside workspace, bootstrapping pip, or discovering escape paths.
  - Kernel (20-24): agents stopped from raw socket calls, reading credentials,
    mutating host filesystem, or inheriting leaked env vars.
  - Cross-tier (25-28): workspace policy consistent across both tiers,
    re-entry safe, graceful degradation works.

  Resolved Issues (2026-04-19)

  Issues found during TESTPLAN audit (2026-04-17) and consumer workflow
  verification (2026-04-19). All code bugs fixed; test bugs incorporated
  into the test plan above.

  ── CODE BUGS (FIXED) ──────────────────────────────────────

  [B1] FIXED (commit a9b5019): `sandflox elevate` re-entry check used
       SANDFLOX_ENABLED instead of SANDFLOX_SANDBOX. Changed to check
       SANDFLOX_SANDBOX, which is set only at the syscall.Exec boundary.
       SANDFLOX_ENABLED remains informational.

  [B2] FIXED (commit a9b5019): elevateExec cacheDir mismatch.
       Resolved cacheDir is now passed from runElevateWithExitCode into
       elevateExec instead of recomputing from projectDir.

  [B3] FIXED (commit 2b8dc4b): `sandflox elevate` had no embedded policy
       fallback. Consumer envs have no policy.toml on disk. elevate now
       falls back to DefaultPolicy() (same as prepare).

  [B4] FIXED (commit 2b8dc4b): sandbox-exec not found via exec.LookPath
       inside restricted PATH. Changed to absolute /usr/bin/sandbox-exec.

  [B5] FIXED (commit 2b8dc4b): entrypoint.sh used bare rm/mkdir/ln for
       symlink bin setup. When sourced by `sandflox elevate` inside a
       restricted PATH, these commands were not found. Changed to use
       ${FLOX_ENV}/bin/ prefixed absolute paths.

  [B6] FIXED (commit 164fb66): .flox/pkgs/sandflox.nix fileset was missing
       policy.toml and requisites*.txt, causing flox build to fail on
       go:embed directives.

  ── TEST BUGS (INCORPORATED) ──────────────────────────────

  [T1] Tests 13, 17: Replaced `which` with `command -v` (bash builtin).

  [T2] Test 15: Replaced `touch` with wrapped commands (mkdir, tee).
       touch is NOT wrapped by fs-filter.

  [T3] Test 15: Replaced `echo bad > policy.toml` with `tee policy.toml`
       (shell redirections bypass fs-filter).

  [T4] Test 15: Replaced `echo test > ./testfile` with `tee ./testfile`.

  [T5] Test 25: Replaced `cp /etc/hosts /tmp/hosts-copy` with
       `cp /etc/hosts ~/.ssh/stolen` (denied path, blocked at both tiers).

  [T6] Test 25: Replaced `touch .git/pwned` with `tee .flox/env/pwned`.

  [T7] Test 15: Used .flox/env/ instead of .git/ for read-only path tests
       (.git/ may not exist in consumer env).

  [W1-W4] All wording/expect comments updated to match actual output format.

  ── DESIGN DECISIONS ──────────────────────────────────────

  [D1] `touch` not added to WriteCmds. Kernel tier catches it. Shell-only
       users accept this gap. Adding it would increase wrapped command count
       for marginal benefit since kernel enforcement is the recommended tier.

  [D2] Shell redirections are inherently unwrappable — documented in test
       plan header and as rationale for the elevate workflow.

  [D3] .git/ path kept in default policy (useful for developer projects).
       Test plan uses .flox/env/ for read-only tests since it always exists.
